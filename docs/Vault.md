# Secret Management — HashiCorp Vault

Step 20 moves the credentials the two Spring Boot services use out of a plain file and behind an
authenticated, audited, revocable secret store.

[`docs/Security.md`](Security.md) named this as the gap before it was closed: every credential lived
in `docker/compose/.env`, and the closing recommendation was to move them to a secret manager. This
is that move — and §11 is honest about how much of it is left.

---

## 2. The dividing line

Consul KV already externalises configuration. Vault is not a second Consul, and the split is
absolute:

> **Configuration is a value that could appear in a screenshot. A secret is a value that could not.**

A port number, a pool size, a retention window, a sampling rate → Consul KV.
A password, a token, an access key → Vault.

Nothing lives in both. A value duplicated across two stores has two sources of truth and will
eventually disagree, and the copy that wins will be whichever one nobody checked.

See [`docs/Consul.md`](Consul.md) for the other half.

---

## 3. Vault runs a real server, not dev mode

`vault server -dev` auto-unseals, keeps everything in memory and prints a fixed root token. It
removes the three things that are actually hard about operating Vault — seal state, the unseal
ceremony, and token lifecycle — which are exactly what this step exists to teach.

So this Vault uses file storage, real initialisation and real unseal. **It boots sealed on every
start**, not just the first.

That cost is paid deliberately and it is not small: the stack no longer comes up from containers
alone. `./scripts/infra.sh up` starts Vault first, unseals it, seeds it, and only then brings up
everything else — see `bootstrap_vault` in [`scripts/infra.sh`](../scripts/infra.sh).

Two details worth knowing before they cost an afternoon:

- **Storage is at `/vault/file`, not a path of our choosing.** The image creates that directory owned
  by the uid the Vault process drops to. A named volume mounted anywhere else is created by Docker as
  root, and Vault fails initialisation with `failed to persist keyring: permission denied` — which
  reads like a storage bug and is a uid.
- **The container command is `server`, with no `-config`.** The image entrypoint already appends
  `-config=/vault/config`. Passing the file explicitly loads the directory *and* the file, defining
  the listener twice, and Vault dies with `address already in use` on its own port — which reads like
  a host port clash and is not one.

The container also needs 512M rather than the 256M it appears to use. Vault mlocks its memory so
secrets are never paged to disk, and mlock reserves its address space up front. At 256M it is killed
during startup, *before* it opens its log — so the container exits 137 with an empty log and no
stated reason. `docker inspect --format '{{.State.OOMKilled}}' lab-vault` is the only place the cause
appears.

## 4. The bootstrap problem

Vault holds the credentials. Something has to hold the credential to Vault.

| File | Holds | Committed |
| --- | --- | --- |
| `docker/compose/.vault-keys.json` | 5 unseal key shares (threshold 3) and the initial root token | **No** — git-ignored, `0600` |
| `docker/compose/.env.vault.order-service` | that service's AppRole `role-id` and `secret-id` | **No** |
| `docker/compose/.env.vault.inventory-service` | the same, for the other service | **No** |

The chain of trust still terminates in a file on disk. Vault did not eliminate that problem; it
**moved and shrank** it. What actually improved:

- one file instead of twenty credentials scattered across `.env`
- what it opens is **scoped per service** — order-service's token cannot read inventory-service's path
- every access is **audited**, including the refusals
- the database credential is **revocable**, and revoking it drops the PostgreSQL user immediately

A reader who concludes that Vault removed the last secret on disk has learned the wrong lesson. In a
real deployment the unseal keys go to separate key holders or a cloud KMS auto-unseal, and the
`secret-id` is delivered short-lived and single-use through a trusted broker. Neither is done here,
and §11 says so again where it belongs.

## 5. Authentication and policy

Each service authenticates with its own **AppRole**: `role-id` identifies who, `secret-id` proves it.
A shared token would be a credential with no identity attached — untraceable in the audit log and
impossible to revoke for one service without revoking it for all of them.

Each AppRole maps to a policy scoped to the paths that service actually reads:

| | `secret/data/application` | `secret/data/order-service` | `secret/data/inventory-service` | `database/creds/order-service` |
| --- | --- | --- | --- | --- |
| `order-service` | read | read | **denied** | read |
| `inventory-service` | read | **denied** | read | **denied** |

**The denial is as much the point as the grant.** A policy nobody has tested a denial against is
indistinguishable from a policy of `*`:

```bash
./scripts/vault.sh deny
```

It authenticates as `order-service`, reads its own path (permitted), reads `inventory-service`'s path
(denied), and tells you where the refusal is recorded.

### The profile-path trap

Spring Cloud Vault reads `<backend>/<context>/<profile>` as well as `<backend>/<context>`. A service
running with `SPRING_PROFILES_ACTIVE=dev` asks for `secret/data/order-service/dev` whether or not
that key exists.

Vault answers a read the token has no grant for with **403, not 404** — so a missing *grant* and a
missing *secret* are the same response. Without the `secret/data/order-service/*` line in the policy,
startup fails with `Status 403 Forbidden [secret/data/order-service/dev]`, naming a key that was
never meant to exist.

## 6. Static secrets — KV v2

The engine is mounted at `secret`, but a read goes to `secret/data/<name>`. The `data/` segment is
not part of the name; it is how the v2 API separates a secret's value from its metadata. A policy
written against `secret/<name>` silently grants nothing.

| Path | Keys |
| --- | --- |
| `secret/application` | `spring.data.redis.password` |
| `secret/order-service` | `spring.flyway.user`, `spring.flyway.password`, `app.storage.minio.access-key`, `app.storage.minio.secret-key` |
| `secret/inventory-service` | `spring.datasource.username`, `spring.datasource.password` |

**The keys in Vault are Spring property names.** There is no mapping layer, so `spring.data.redis.password`
in Vault becomes exactly that property. A typo produces a silent fallback to the `application.yml`
default rather than an error — which is the failure mode §9 exists to catch.

## 7. Dynamic PostgreSQL credentials

This is the part that makes Vault something other than an encrypted `.env`.

Reading `database/creds/order-service` causes Vault to **create a real PostgreSQL user** with a
one-hour lease, renew it while the service runs, and drop it when the lease ends.

```bash
./scripts/vault.sh creds     # issue one and look at it
./scripts/vault.sh leases    # what is outstanding
./scripts/vault.sh revoke    # drop them all, now
```

### The import line is not ordinary

```yaml
spring:
  config:
    import: "optional:consul:,optional:vault://,optional:vault://database/creds/order-service?prefix=spring.datasource."
```

The `prefix` is load-bearing. Vault's database engine answers with two keys called `username` and
`password`. Without a prefix they arrive as **root-level** properties of exactly those names — a trap
on two counts. They never reach `spring.datasource.*` at all, and `username` collides with the
`USERNAME` environment variable that Windows sets on every machine. `systemEnvironment` outranks every
config-data import, so on a developer's laptop the datasource would try to log in as the person using
it.

This is **not** `spring.cloud.vault.database.username-property`. That property pair belongs to the
older bootstrap-context mechanism and does nothing under the config-data API used here: the request
is never made and the datasource silently keeps its static fallback. The difference is visible only
in `/actuator/env`, which is where it was confirmed.

### Flyway keeps a static credential

Migrations run DDL, and every object created acquires an owner. A short-lived dynamic user would own
the schema and then be dropped at the end of its lease — taking the schema with it, or failing the
drop. Either way the failure appears an hour after the deployment that caused it.

So `spring.flyway.user` / `spring.flyway.password` come from KV v2 and name the **owning role**, and
only the runtime Hikari pool uses dynamic credentials. This split is what real deployments do.

### Grants, and why there is a group role

`seed_postgres_role` in `scripts/vault.sh` creates a `NOLOGIN` group role `order_readwrite` and grants
it the runtime privileges, then sets `ALTER DEFAULT PRIVILEGES` so objects a *future* migration creates
are covered too. Vault's creation statement then just does `IN ROLE order_readwrite INHERIT`.

Spelling the grants out in the creation statement instead would work exactly once: the grant is
evaluated when the user is created, so any table added by a later migration would be invisible to
every credential issued before it.

### Lease TTL and the connection pool

| Setting | Value | Where |
| --- | --- | --- |
| Lease default TTL | `1h` | `DB_DEFAULT_TTL`, `scripts/vault.sh` |
| Lease max TTL | `8h` | `DB_MAX_TTL` — capped by `max_lease_ttl` in `vault.hcl` |
| Hikari `max-lifetime` | `15m` | `application.yml` |

**`max-lifetime` must stay below the lease TTL.** A pooled connection authenticated once and is not
re-checked afterwards. If the credential outlived the pool's recycling interval, the pool would keep
handing out connections belonging to a PostgreSQL user that no longer exists, and the failure would
surface as a slow trickle of errors minutes after the expiry rather than at the moment anything
changed. Recycling well inside the lease means a dead credential is discovered by the pool, on its own
schedule, instead of by a user.

Raising `max_lease_ttl` in `vault.hcl` without raising the role's `max_ttl` does nothing; lowering it
below the role's silently truncates the role, and Vault warns rather than refusing.

## 8. Why the Inventory Service does not get dynamic credentials

Oracle is not one of the database plugins built into the Vault binary. It is distributed separately as
`vault-plugin-database-oracle`, needs the Oracle Instant Client libraries present, and must be placed
in a plugin directory and registered in the plugin catalogue.

That is not done here. `inventory-service` therefore reads a **static** Oracle credential from KV v2
while `order-service` is handed a generated one.

This is a stated limitation, not an oversight. The stack does not run entirely on dynamic credentials
and this document does not claim it does.

## 9. Prove it came from Vault

This is the check that matters, because a service that silently fell back to its `application.yml`
defaults looks **identical** to one correctly wired to Vault — the defaults are the working lab
credentials.

```bash
./scripts/vault.sh whoami
```

```
PostgreSQL sessions on orderdb, by user:

user  sessions
postgres  2
v-approle-order-se-Bo87j7b3dFzlJYLRwvOo-1784889461  2
```

A user named `v-approle-order-se-...` is a credential Vault minted, and the `approle` in the middle
records which auth method asked for it. Seeing `order_user` instead means the service is **not** using
Vault.

Two supporting checks:

- `./scripts/vault.sh audit` — every read, by path, including refusals
- both services set `spring.cloud.vault.fail-fast: true`, so an unreachable Vault stops startup rather
  than letting the fallback take over quietly

## 10. Observability

Vault is infrastructure here, so it gets the same treatment as everything else. A secret store that
fails silently is worse than a file.

### Metrics

Scraped at `/v1/sys/metrics?format=prometheus`, without a token — the listener sets
`unauthenticated_metrics_access`. The alternative is putting a Vault token in the monitoring config in
order to monitor the credential store, and having the metrics pipeline break the day it lapses. The
endpoint exposes seal state, request rates, lease and token counts; no secret material.

### The `vault_core_unsealed` trap

Vault publishes this gauge **twice** — once labelled with the real cluster id, and once with
`cluster=""` permanently at `0`:

```
vault_core_unsealed{cluster=""} 0
vault_core_unsealed{cluster="vault-cluster-4ab09250"} 1
```

So the obvious `vault_core_unsealed == 0` matches on a perfectly healthy Vault and fires forever,
which trains everyone to ignore the one time it means something. Every alert and panel here uses
`max(vault_core_unsealed)`.

### Alerts

| Alert | Severity | Fires when |
| --- | --- | --- |
| `VaultSealed` | critical | `max(vault_core_unsealed) == 0` for 1m |
| `VaultDown` | critical | the scrape target is down for 1m |
| `VaultLeaseCountCollapsed` | warning | no outstanding leases while services are up, 5m |
| `VaultPermissionDenied` | warning | sustained 403s for 5m |
| `VaultAuditDeviceFailing` | critical | audit log write failures |

`VaultPermissionDenied` is built on `vault_core_response_status_code{code="403"}` and **not** on
`vault_token_create_failed`, which is the obvious choice and is wrong: that series does not exist
until the first failure, so an alert on it cannot be tested and would sit dead in the rules file
indefinitely.

### Audit log

Enabled as a file device at `/vault/logs/audit.log` on its own volume, tailed into Loki by the
`vault-audit` Promtail job.

Its own volume and its own job, for two reasons: Vault runs as a different uid and could not write
into the services' log directory, and its records have a completely different schema. Run through the
service pipeline, every extracted field would be null and every label empty — which looks ingested
rather than broken.

Safe to ship, because Vault HMACs every secret value before writing. The log records *that* a value
was read, never the value.

`request.path` is deliberately **not** promoted to a Loki label: it carries a lease id on every
dynamic credential renewal, so labelling it would create a stream per credential issued and bring the
ingester down. Filter with `| json | request_path=~"..."` instead.

> **Vault refuses every request when all audit devices are failing.** That is deliberate — an
> unauditable secret store is considered worse than an unavailable one — and it is startling the first
> time, because the symptom is a total Vault outage and the cause is a full disk.

## 11. What this step did **not** do

Vault can only serve a secret to something that can ask for it.

The two Spring Boot services can ask. **Keycloak, Grafana, Redis, MinIO, PostgreSQL and Oracle cannot** —
they read environment variables handed to them at boot, and no amount of Vault configuration changes
that. Their bootstrap credentials are still in `docker/compose/.env`.

| Still in `.env` | Why |
| --- | --- |
| `POSTGRES_SUPERUSER_PASSWORD` | it is what Vault itself uses to reach PostgreSQL and create users |
| `ORACLE_SYS_PASSWORD`, `KEYCLOAK_*`, `GRAFANA_ADMIN_PASSWORD`, `MINIO_ROOT_*`, monitoring accounts | container bootstrap, read at boot from the environment |

Closing that gap needs **Vault Agent** rendering an env file per container — a different mechanism with
its own failure modes. It is a candidate for a later step and was not attempted here.

Also deliberately absent, and all of them things a real deployment would do:

- **No TLS.** Unseal keys, the root token and every secret cross this listener in cleartext. The port
  binds to loopback and nothing outside `lab-net` reaches it, which is the only reason this is
  tolerable. In production, TLS here is not optional — the thing being protected is the protection
  mechanism itself.
- **`secret_id_ttl=0`** — the AppRole secret-id never expires. A real deployment issues a short-lived,
  single-use secret-id through a trusted broker.
- **Unseal keys in one file**, all five shares together. Shamir splitting is theatre when one file
  holds every share; the ceremony is what is being taught.
- **No auto-unseal**, so a machine reboot leaves the stack down until someone runs the script.
- **Root token retained** after seeding rather than revoked.

## 12. Commands

```bash
./scripts/vault.sh bootstrap   # init + unseal + seed, idempotent
./scripts/vault.sh status      # seal state and mounts
./scripts/vault.sh unseal      # after any restart
./scripts/vault.sh creds       # issue a dynamic credential and show it
./scripts/vault.sh leases      # what is outstanding
./scripts/vault.sh revoke      # revoke them all
./scripts/vault.sh whoami      # which PostgreSQL user the service is actually using
./scripts/vault.sh deny        # prove the policy boundary
./scripts/vault.sh audit 50    # tail the audit log

./scripts/chaos.sh vault seal    # inject: secrets unreadable
./scripts/chaos.sh vault revoke  # inject: drop the dynamic credentials
./scripts/chaos.sh reset         # unseals, among everything else
```

The Vault UI is at `http://127.0.0.1:8200`. The token:

```bash
jq -r .root_token docker/compose/.vault-keys.json
```
